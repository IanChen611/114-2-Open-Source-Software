## 1. 專題概述

本專題提案旨在設計並實作一款**以 Linux 核心模組驅動的數獨遊戲**。遊戲的核心邏輯——包含謎題生成、落子驗證及解答確認——皆實作於核心空間，並透過字元裝置（`/dev/sudoku`）對外暴露介面。使用者空間的輕量命令列客戶端藉由讀寫此裝置，提供互動式的數獨遊戲體驗。

本專題整合課程要求的兩大技術核心：

| 技術核心 | 在本專題中的角色 |
|---|---|
| **Git** | 版本控管、分支工作流程，以及透過 Git Hook 自動執行 CI |
| **核心程式設計** | Linux 核心模組實作所有數獨核心邏輯 |

## 2. 動機

核心程式設計的入門教材通常只示範「Hello, World」模組，雖能演示模組的載入與卸載，卻缺乏實際應用意義。本專題希望在核心層級實作真正具有互動性的程式，使學習過程更具體且深刻。

選擇數獨作為題目的理由如下：

- 驗證與生成演算法（回溯法、約束傳播）具備一定的演算法深度，同時複雜度有界，適合在核心中實作而不至於造成系統不穩定。
- 遊戲狀態天然地對應到固定大小的核心資料結構（9×9 整數陣列），使記憶體管理更加單純。
- 互動模型——使用者送出落子、核心驗證後回傳結果——與字元裝置的 `read`/`write` 介面高度吻合。

## 3. 系統架構

```
使用者空間
┌─────────────────────────────────┐
│  sudoku-cli  (C 程式)           │
│  - 渲染棋盤                     │
│  - 讀取使用者輸入                │
│  - 呼叫 write(fd, move, ...)    │
│  - 呼叫 read(fd, state, ...)    │
└────────────┬────────────────────┘
             │  open("/dev/sudoku")
             │  write / read / ioctl
─────────────┼──────────────────── 核心邊界
             ▼
┌─────────────────────────────────┐
│  sudoku.ko  (核心模組)          │
│                                 │
│  ┌─────────────┐                │
│  │ 謎題儲存區   │ (9×9 棋盤、    │
│  │             │  解答遮罩)     │
│  └──────┬──────┘                │
│         │                       │
│  ┌──────▼──────┐                │
│  │  驗證器     │ (行、列、宮格  │
│  │             │  約束檢查)     │
│  └──────┬──────┘                │
│         │                       │
│  ┌──────▼──────┐                │
│  │  生成器     │ (回溯法        │
│  │             │  + 隨機種子)   │
│  └─────────────┘                │
└─────────────────────────────────┘
```

### 3.1 字元裝置介面

核心模組動態申請主裝置號並註冊字元裝置，支援三種操作：

- **`write`** — 以 `列 欄 數值` 格式送出落子指令（例如 `"3 5 7"`）。核心驗證是否符合數獨約束，合法則更新棋盤。
- **`read`** — 回傳目前的 81 位元組棋盤狀態（每個位元組為數字 1–9 或 0 代表空格），以及一個狀態位元組（勝利 / 進行中 / 非法）。
- **`ioctl`** — 控制指令：`NEW_GAME`（生成新謎題）、`RESET`（還原初始謎題）、`HINT`（提示一格正確答案）。

### 3.2 核心模組內部實作

模組在核心記憶體中維護以下陣列：

```c
static uint8_t board[9][9];     // 目前遊戲狀態
static uint8_t solution[9][9];  // 預先求解的參考答案
static uint8_t fixed[9][9];     // 謎題初始給定格（唯讀）
```

謎題生成採用有隨機種子的回溯演算法。由於核心空間無法使用 `rand()`，模組改以 `<linux/random.h>` 中的 `get_random_u32()` 作為亂數來源。求解器以迭代回溯加上約束檢查填滿整個棋盤，再移除可設定數量的格子（預設為 40 格）以生成謎題。

## 4. Git 工作流程

所有開發遵循結構化的 Git 工作流程：

- **`main`** — 穩定分支，任何時候都必須可以成功建置。
- **`dev`** — 整合分支，用於合併已完成的功能。
- **`feature/<名稱>`** — 個別功能分支（例如 `feature/generator`、`feature/validator`、`feature/cli`）。

### 4.1 Git Hook

**pre-commit hook** 放置於 `.git/hooks/pre-commit`，每次提交前自動執行使用者空間的單元測試框架。該框架將核心數獨邏輯（獨立編譯為靜態函式庫，與模組膠合程式碼分離）連結後執行驗證測試；若任何測試失敗，提交將被拒絕。

## 7. 技術風險與因應措施

| 風險 | 因應措施 |
|---|---|
| 模組中的錯誤指標導致核心崩潰（Kernel Panic） | 在 QEMU 虛擬機內開發與測試，絕不在實體主機上直接測試 |
| `get_random_u32()` API 在不同核心版本間的差異 | 鎖定特定 LTS 核心（6.1）進行開發，並於文件中記錄版本需求 |
| 阻塞式 `read` 導致使用者行程停頓 | 採用非阻塞讀取模型，立即回傳目前狀態 |
